## Auth model

- **Backend** (`app.py:56-68` middleware + `autosentinx/security.py`): JWT **bearer** tokens (HS256, `sub=email`,
  `exp=jwt_expire_hours`). A global `http` middleware rejects any request **without** `Authorization: Bearer <jwt>`
  with `401 {"detail": ...}`, **except** the open set: `/`, `/health`, `/auth/signup`, `/auth/login`, `/docs`,
  `/redoc`, `/openapi.json`, `/favicon.ico`, and any path starting with `/docs`, `/redoc`, `/static`.
- **Frontend** uses a **BFF / proxy** pattern. The browser never holds the JWT:
  - `app/api/auth/login/route.ts` calls backend `/auth/login` (falls back to `/auth/signup` on 401), then stores the
    JWT in an **httpOnly cookie** `sx_jwt` (24h).
  - `app/api/[...path]/route.ts` proxies every `/api/<path>` browser call to `BACKEND/<path>`, reading `sx_jwt` from
    the cookie and attaching `Authorization: Bearer <jwt>`. So all backend auth is satisfied server-side.
  - `app/api/auth/logout/route.ts` just deletes the cookie (no backend call).
- `BACKEND_BASE` env (default `http://127.0.0.1:8080`) is the proxy target.

### Console view-model shapes (the FE-facing contract — `autosentinx/console.py` → `sentinx-web/lib/types.ts`)

- **Run summary row** (`run_summary`, used by #17 and embedded in #18):
  `{id, targetUrl, agentName, status, startedAt, operator, playsTotal, playsDone, note}`.
  `status` collapses `pending_approval` → `"running"`. `agentName` is **hardcoded** to
  `"AARAV — NBFC voice debt-collection agent"`.
- **Full run** (`run_full`, #18): summary fields **+** `{endedAt:null, engineVersion, scenarioLibVersion,
  durationSec:null, observations[]}`.
- **Observation** (camelCase): `{id, objectiveSlug, objectiveId, title, description, module(security|compliance),
  severity, oracle(panel|single-judge|fairness), rawOutcome, verdictScore, judgeVotes[], detectorHits[], crosswalk[],
  bypass, incidentId, pairedId, evidence[], numTurns, detectedIn, reproduced, technique, persona}` (+ `fairness{…}` for
  fairness rows). The FE derives the product `outcome` (FAIL/RISK/PASS) itself in `lib/outcome.ts`; the backend emits
  only `rawOutcome` + inputs.

## Frontend calls (what the browser actually invokes)

All go through the same-origin BFF; the proxy strips `/api` and forwards the rest to the backend.

| FE call (`lib/api.ts` unless noted) | Browser request | → Backend endpoint | Expected response |
|---|---|---|---|
| `login(email,password)` | `POST /api/auth/login` JSON `{email,password}` | `/auth/login` → fallback `/auth/signup` | `{email}` (token kept server-side in cookie) |
| `logout()` | `POST /api/auth/logout` | — (cookie delete only) | `{ok:true}` |
| `startScan(cfg)` | `POST /api/scan?strategy=ucb&budget=<n||6>` | `POST /scan` | `{run_id, status}` |
| `approveScan(runId)` | `POST /api/runs/<id>/approve` | `POST /runs/{id}/approve` | (ignored body) |
| `getRun(runId)` + `useRun` poll | `GET /api/console/runs/<id>` | `GET /console/runs/{id}` | full run view-model → `withOutcomes()` |
| `listRuns()` | `GET /api/console/runs` | `GET /console/runs` | `{runs:[…summary…]}` |
| `logger` (`lib/logger.ts`) | `POST /api/log` | — (local Next route, not proxied) | `204` |

`app/new/page.tsx` is the only feature page calling `startScan`/`approveScan`; `app/login/page.tsx` calls `login`;
`app/runs/[id]/**` pages all read through `useRun → getRun`.

F4 is zero-runtime doc drift → LOW. F5 is latent today (FE approves before navigating) but is the only gap with a real runtime-hang mode → MED. `pl2-plan.md §(b)` gap #N = finding F⟨N⟩ here._

### F1 — HIGH · `startScan` drops the operator's `endpoint` + `agentName`
- **Gap:** `lib/api.ts startScan(cfg)` accepts `{endpoint, agentName, budget}` and `app/new/page.tsx:29` passes a user-typed endpoint + agent, but the function serializes only `strategy=ucb&budget=…`. The backend `POST /scan` has **no** endpoint/agent field — the target is hard-wired to `aarav_base_url` (`target.py`; DECISIONS D1/D-Q14; BACKEND-UPDATE.md:159).
- **Impact:** the "Target API endpoint" + "Agent name" form fields are **UI theater** — operator input has zero backend effect. The "Endpoint reachable" success copy (`new/page.tsx:110`) is asserted with **no** reachability check.
- **Owner / action:** Product decision, then FE-now / BE-later. **Now (FE, Phase 4):** make copy honest — present the target as "configured target (fixed in this build)", stop asserting unverified reachability. **Later (BE):** add `endpoint`/`agent` to `POST /scan`.

### F2 — HIGH · displayed agent name is hardcoded
- **Gap:** `console.py:_agent_name` always returns `"AARAV — NBFC voice debt-collection agent"`, regardless of input.
- **Impact:** even the **read path** can't reflect operator input (same root as F1).
- **Owner / action:** BE-later (derive agent from the run) / FE-now (don't imply the displayed name came from operator input).

### F3 — MED · budget default drift (FE 6 vs BE 40)
- **Gap:** `app/new/page.tsx` calls `startScan` with no budget → `lib/api.ts` defaults **6**; backend `/scan` default is **40**. FE always sends 6, so the BE default never applies; on-screen "one evaluation run" maps to neither.
- **Owner / action:** FE + docs. Pick one intended budget, set it explicitly in `lib/api.ts`, state it in `/new` copy, record it here. No BE change.

### F4 — LOW · poll-target doc drift
- **Gap:** BACKEND-UPDATE-2 §h / DECISIONS D-Q15 say the UI polls `/runs/{id}`; the FE actually polls **`/console/runs/{id}`** (`use-run.ts`→`getRun`). The console endpoint is the better, view-model-shaped choice (and reflects the `pending_approval→running` collapse).
- **Owner / action:** Docs only. Update BACKEND-UPDATE-2 §h/§82 + D-Q15 to name `/console/runs/{id}` as the poll target. (Not a bug.)

### F5 — MED · `pending_approval` masking can hang the poller
- **Gap:** `console.py:run_summary` collapses `pending_approval` → `"running"`; `useRun` stops only when `status !== "running"`. A run created but **never approved** reports `"running"` forever with `playsDone=0` → the poller never terminates. Latent today (the FE calls `approveScan` immediately).
- **Owner / action:** FE-defensive (Phase 5) + BE-clean. **FE:** add a poll-guard (max-duration / `playsDone===0` stall timeout / "awaiting approval or stalled" state). **BE:** expose `pending_approval` distinctly so the FE can stop correctly.

### F6 — LOW · 14 of 21 endpoints are unconsumed by the FE
- **Gap:** `/auth/me`, `/health`, `/catalog*`, `/techniques*`, `/coverage`, `/selection/stats`, `/ingest`, `/audit`, `/runs*`, `/runs/{id}/transcript` have no FE consumer. `/console/runs/{id}` pre-joins catalog/crosswalk, satisfying the BACKEND-UPDATE `/catalog/{slug}` recommendation a different way.
- **Owner / action:** Backlog (no change). Governance surfaces (`/audit`, `/coverage`, `/selection/stats`) are future-feature candidates.

### F7 — LOW · BFF proxy only handles GET/POST
- **Gap:** `app/api/[...path]/route.ts` exports only `GET` and `POST`. Matches the backend (GET/POST only) today; any future `PUT`/`DELETE`/`PATCH` route is unreachable (405).
- **Owner / action:** Forward-looking note. Extend the proxy when a new verb is added.

### F8 — LOW · `approveScan` returns `void` (detail is shown today, just not branchable)
- **Gap:** `approveScan` (`sentinx-web/lib/api.ts:45-47`) returns `void` — it never returns the parsed body, so callers can't branch 404-vs-409 programmatically. The thrown `Error.message` *does* carry the backend `.detail` (the shared `req()` helper extracts it at `api.ts:11`), and **main's caller surfaces it** (`app/new/page.tsx:44` shows `e2.message`), so the operator does see the specific 409/404 reason today. The residual gap is structural/fragile — any future caller that catches without reading `.message` loses the reason — plus the detail isn't shown in a dedicated in-dialog `aria-live` region.
- **Owner / action:** FE-small (Phase 4, P4-4). Keep `req()`'s `.detail` extraction; surface it in a dedicated in-dialog approve-error region (`aria-live`); optionally have `approveScan` return the parsed body so callers can branch. (Note: the old "swallows the detail" framing overstated it — main already propagates `e.message`.)
