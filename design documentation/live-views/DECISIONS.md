## V2 Arena (Q6+)

- **D-LV13 · V2 = the Product Admin's LIVE window into a running audit.** *Supersedes the earlier "separate QA instrument / QA-first" wording (2026-06-19).* There is **no separate QA tool**; the admin watches and reviews the run *through V2 itself*. It must surface trustworthy depth (coverage, judge quality, degraded-run clarity, where the agent broke) **as part of** the live duel, not as a different mode. *(Q6, revised)*
- **D-LV15 · V2 spine = the turn-by-turn "frame-ladder" readout.** Each play is a strip of per-turn cells (held / wavered / gave-the-line, from the real `turns[].label`), capped by the ruling; momentum is the **shape of the label sequence**, **not** a depth/distance axis. Rationale: on real runs every attack reaches its final phase (`arcComplete` for holds AND the breach), so a depth axis renders all plays identical and misleads ("full bar = attacker won"); the label-shape is the only honest discriminator, and the breach reads as a visibly different ribbon that **forks** at the verdict cap. Adopts the "Ladder/Telegraph/Overturn" concept (`v2-concept-LTO.md`) as V2's chassis. *(Q8, decided with side-by-side previews)*

- **D-LV14 · MANDATE — reimagine, do not polish.** This exercise must push V2's **abstraction, engagement, and functionality well beyond the current `duel2.html` prototype** — *more* 2D-game-design and abstraction, not less. Keep the **game-design philosophy + duel concepts** (matchup, stakeout map, attack arc, gate-delta climax, …) and render them on the **DESIGN.md calm-instrument foundation** (`00-foundation.md`). The current prototype is the **floor to exceed, not the target**. *(2026-06-19 directive)*

- **D-LV16 · Telegraph IN (V2-internal), intent-led.** Show the attacker's pre-committed **next phase** as a faint "planned next move" ghost. **Honest basis (verified in code):** within a play the phase sequence is **fixed** (`technique.phase_plan`, `library.py`/`runner.py:334`); the attacker only advances through it (Comply / 2-turn timer) and can **stop early** on success → un-played phases read **"not reached"**, never "would have won". What adapts: per-turn **wording** + per-**play** technique (UCB bandit *across* plays, `runner.py:185/200`), **not** the within-play sequence. **Caveat:** if the engine ever re-plans within a play, the ghost relabels "planned next" → **"current plan (may adapt)"**. IP is a non-issue (internal, `D-LV6`). *(Q9)*
- **D-LV17 · Lead with INTENT, not phase-names.** Phase names (`"Sub-request-1"`, etc.) are **internal metadata, not a signal**. V2 surfaces the plain-language **`intent`** everywhere (arc bands, telegraph, captions); the raw phase-name is hidden/secondary. A general V2 rule. *(Q9 — owner point)*
- **D-LV18 · One orchestrated moment = the verdict-cap overturn; replay is a user-invoked function.** The overturn settle is the single LV-4 "delight" beat; **replay-to-pivot is a button** (not a second auto beat). Default = single composed settle; the 2-beat "agent's gate said *clean* → fork to BREACHED" plays **only** on a real `gateDelta.disagree`. *(synthesis Q3)*
- **D-LV19 · Fork legibility.** The self-report "clean" cap renders **visibly subordinate/advisory** ("agent's own gate"), never mistaken for a second real verdict; **"The agent believed it held. The judges disagree."** stays the single hero line. *(synthesis Q4)*
- **D-LV20 · Clean & degraded reads.** A zero-breach run reads as a confident **"cleared" board** (composed HeldState + ★ critical-held); the errored/blocked play is a **first-class dashed "not assessed" ribbon + a CRITICAL-untested coverage flag**, never softened. *(synthesis Q5)*
- **D-LV21 · V2 theme = follow `../DESIGN.md` (`D-Q6`).** Light default + dark first-class toggle, respecting `prefers-color-scheme` on first load; both fully designed. **Not** dark-only/dark-default — one theme system across live views + report. *(Q10)*

## V3 Forensic (Q11+)

- **D-LV22 · V3 = the V2 play "fully expanded" + two light verdict-trust tools.** Internal-unrestricted full detail: complete turn-by-turn **transcript**, **all judge votes (model names + full reasons + specificity)**, all **detectors**, full **recon profile**, **run provenance**, and **debug internals** (belief state, why each phase advanced, early-stop reason). Reached by rolling **down** from a V2 play (or the internal cross-link from a report finding). **Plus two light tools** that serve the admin's "is-this-verdict-trustworthy" review: **Re-judge** (verdict stability/reproducibility) and **Judge-diff inspector** (the raw judge prompts/responses/rubric → why one committed and others didn't). Customer never sees V3 (`D-LV6`). *(Q11)*

- **D-LV24 · Run-level layout = two pillar bands; dual-duty = two linked ribbons.** Stack the ladder in **Security** (top) + **Compliance** (bottom) bands, each with its real **withstood-fraction** (`D-Q11`), plays **severity-ordered** within; coverage line + recon prelude above. A **dual-duty attack** (bypasses a guardrail AND produces a harm) shows as **two linked ribbons — one per band — the same conversation, two findings**, sharing an **incident id + paired-link connector**, each marking **its pillar's breach-point** (bypass turn / harm turn), counting in **both** withstood-fractions, cross-linking to **both** report observations — mirroring the report's two-row model (`D-Q2`/`D8`). **DB-backed** (runs/attempts/turns persist in Neon; the shared id + per-pillar observations are what `D8` emits). **Today** (`D8` unbuilt) every play is single-pillar = one ribbon in one band; the **paired-twin activates when `D8` lands** (render the contract now, degrade gracefully). *(Q13)*

- **D-LV25 · Run-Intensity "effort" dial on Run Config (C3).** A 6-level control `[ low · med · high · xhigh · max · ultra ]` (Claude-Code-effort style) set at run start, scaling three honest levers: **attacks** (campaign `budget` = # plays, `runner.py:146`), **turns/attack** (`max_turns`), and **breadth** (technique×persona variants per objective). **Judges fixed across levels** (constant grading rigor for fair comparison). **`med` = doubled baseline (16 attacks / 16 turns)** — the F3 "double both"; both pillars covered (bandit selects across the 4 Security + 33 Compliance objectives). Proposed mapping (tunable): low 8/8/1 · med 16/16/1 · high 28/18/2 · xhigh 40/20/2 · max 60/24/3 · ultra full-catalog×all-techniques/30/all. Chosen level shown in **RunProvenance** + on V2, with a live **estimate** (duration / LLM-call budget). **Canonical edit:** add the dial to **C3 Run Config**; engine drives `run_budget(budget=…)` + `max_turns`. *(Q14 — owner directive)*

## Build sequencing & the engine port (D-LV-dep3)

- **D-LV26 · Engine port (`D-LV-dep3`) is PARKED until the `RunView` contract is locked — and the contract is NOT lockable yet because none of V1, V2 or V3 is finalized.** *(decided 2026-06-20)*
  - **The dependency:** `ENGINE-PORT-PLAN.md` treats `sentinx-web/lib/runview.ts` as its **frozen target**. A port to a contract that can still change is premature. The contract cannot be frozen while a *consuming* view is still in flight.
  - **The fact, recorded plainly — none of the three live views is finalized:**
    - **V1** — **unbuilt** (`D-LV23` defers it; sequence V2 → V3 → V1). Because V1 lives in the **live-duel family** (`D-LV10`), it consumes `RunView` and will **exercise / possibly extend** the shape. It is the principal reason the contract is not yet stable.
    - **V2 + V3** — **built and adversarially scored to 96–97/100 ("shippable")** but **NOT signed-off / finalized**; they may still change. "Scored high" ≠ "frozen."
    - **Open decisions still live:** `OPEN-LV1` (V1 ⇄ C4) and `OPEN-LV2` (V2 demo mode) — either can touch the surface or the shape.
  - **Why parking is low-risk (not wasted):** the doc discipline is **additive, ID-keyed, supersedes-nothing** (`README.md`), so any future shape growth **extends** the port (one projector line per new field) and **never invalidates** it. The plan already paid for itself by **de-risking** the architecture (proved the contract is producible server-side: poll `/console/runs/{id}`, **no core migration**, both-pillar "for free" via `run_budget`) and now stands as a **living ledger**.
  - **Trigger to un-park:** **V1 built → `OPEN-LV1`/`OPEN-LV2` resolved → V2/V3 finalized → `runview.ts` frozen** ⇒ then build the port (Phase 1 → 2 → 3 per `ENGINE-PORT-PLAN.md`).
  - **Notes for whoever builds it later:** (a) the canonical-rule fix (`EP-5`, `outcome.ts → outcome.py`) is **shape-independent** so it *could* land early, but it only bites the borderline single-judge RISK band `[0.3, 0.55)` — captured runs are clear-cut, so it can wait too; (b) reconcile against **`feat/security-governance-keystones`** (it edits `runner.py` and adds an `roe.py` *policy* module — distinct from the `Run.roe` JSON config the port writes; expect a `runner.py` line-anchor shift).
